# Compléxité des Données

***Données simples :*** 0 ou 1, allumé ou éteint, présent ou absent...

***Données complexes :*** distance, température, accélération


Possibilité pour traiter des données complexes :

- Passer en transmission binaire, mais ça prend trop de lignes

![Capture écran](../../../../images/bus.png)

- Problème avec ceci : 
  - 1 capteur => 8 lignes de communication
  - 2 capteurs => 16 lignes
  - 3 capteur => 24 lignes
  - ...

___

Donc plutôt qu'utiliser un fil pour chaque bit qu'on veut transmettre.
On va trouver une manière de raccorder plusieurs composants sur le même fil.

Solution -> **Un bus de données**

- C'est un ensemble de lignes de communication permettant l'échange de données entre plusieurs composants d'un système informatique.

![Capture écran](../../../../images/bus_comm.png)

___

### *Différents Bus*

**Bus parallèle**

- Bus qui transmet plusieurs bits simultanément sur plusieurs lignes de communication.
- Exemples : 
  - bus entre CPU et la RAM d'un ordi.
  - **Schéma :**
    - ![Capture écran](../../../../images/bus_parallèle.png)

**Limites physiques :**

- Nombre de lignes de communication.
- Interférences électromagnétiques à haute fréquence.
- Longueur identique des lignes de communication.

![Capture écran](../../../../images/bus_para.png)

(Pas la meilleure solution non plus)

PCI (exemple de bus parallèle)
___

**Bus série (UART)**

PCIe (exemple de bus série x2/x4/x8/x16, bus de série mis en même temps)

- Bus qui transmet les données bit par bit sur une seule ligne de communication.

- Exemples :
  - USB, SATA, PCIe, i²c, SPI, UART.

![Capture écran](../../../../images/bus_serie.png)

- simplex : un qui transmet un qui reçoit, vont dans un seul sens.
  - Un appareil transmet.
  - l'autre reçoit.
  - pas de retour possible
- Exemple :
  - Clavier -> PC
  - TV (antenne -> TV)


- half-duplex : communication dans les deux sens, mais pas en même temps.
  - Chaque appareil peut : 
    - Transmettre
    - Recevoir
  - Mais pas simultanément.
- Exemple :
  - Talkie-Walkie


- full-duplex : communication dans les deux sens en même temps.
  - Chaque appareil peut :
    - Transmettre
    - Recevoir
    - Simultanément.
  - Exemple :
    - Téléphone
    - Réseau Ethernet moderne (switché)




**Problème du UART**

- **Configuration**

  - À choisir : 
    - **Vitesse de transmission** baud rate = vitesse de transmission des signaux (pas en mesure de décoder le message reçue si pas configuré)
    - **Nombre de bits de données** de 5 à 9 (bus en série 1 par 1, faut savoir quand on a transmit le bit de donnée)
    - **Bit de parité** aucun (N), pair (E) ou impair (O) = détection d'erreur
    - **Nombre de bits de STOP 1 ou 2**
  - ![Capture écran](../../../../images/exemple1.png)

**Codage de données**

![Capture écran](../../../../images/codage_donnees.png)

*Bit poids faible et fort :*

- Exemple : 1011 -> (1 - fort)01(1 - faible)

**/!\ UART commence par le bit de point faible.**

- Donc de droite vers la gauche.

*Le bit de parité (en vert sur le schéma)*

*Pour signifier STOP, donc bit de STOP, état haut*

**Signaux**

Faut maintenant transformer des bits (0 et 1) en signaux électriques.

*Plusieurs standards courants :*

- **TTL** standard pour les communications série sur des courtes distances :
  - ![Capture écran](../../../../images/ttl.png)
  - État haut déterminé, comme pour RPI3 à 3.3V et état bas à 0V.

- **RS-232** standard historique pour les communications série :
  - ![Capture écran](../../../../images/rs-232.png)
  - À l'état bas on ne sait pas où on se retrouve entre -15V et -3V.

- **RS-485** standard pour les communications série en réseau :
  - ![Capture écran](../../../../images/rs-485.png)
  - Avantage de traverser des distances plus longues (câblage en série sur des longues distances).
  - Plus important : pas savoir la tension d'un fil, mais est-ce que c'est le fil + qui a une tension plus elevé que le fil - ou l'inverse. Pour ceci qu'ils se croisent.

### Et si on veut plusieurs destinataires ?

Problème -> UART est limité à *2 participants*.

Pour connecter plusieurs composants sur le même bus, il faut : *Horloge commune* pour synchroniser les participants *adresses* pour identifier le destinataire

-> Solution -> **Bus i²c**

![Capture écran](../../../../images/busi2c.png)

![Capture écran](../../../../images/codagei2c.png)

**Signal d'horloge** -> SCL en haut, c'est quand on va s'intéresser. Uniquement quand SCL est en haut.

**/!\ Pour i²c on commence par le bit poids fort**

- C'est le **Signal d'horloge** qui va donner les informations aux composants. Ils auront tous adapté cette configuration là.

**Exemple :**

- ![Capture écran](../../../../images/exemple2.png)

- Quand c'est en bleu le maître impose à l'esclave.
- Le SDA maître pour le ACK, va envoyer un signal bleu vers le haut, et l'esclave va le tirer vers le bas pour dire qu'il est bien reçu.
- C'est le SDA maître aussi qui va dire le STOP.

#### Quel bus choisir ?

- **UART** communication point-à-point simple.
- **i²c** plusieurs capteurs sur le même bus

Souvent le choix est imposé par les composants que l'on utilise.

#### Et SPI ? (capteur dont on a pas parler)

 - Plus rapide que i²c.
 - Sans adressage natif, nécessite un fil de sélection pour chaque escale.
 - Full-duplex
