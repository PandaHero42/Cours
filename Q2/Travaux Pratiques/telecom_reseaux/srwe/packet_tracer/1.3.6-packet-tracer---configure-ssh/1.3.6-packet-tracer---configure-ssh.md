# Packet Tracer – Configure SSH

> **Objectif** : Remplacer Telnet (non sécurisé, données en clair) par SSH (données chiffrées) pour administrer un switch à distance.

---

## Topologie

| Device | Interface | Adresse IP  | Masque        |
|--------|-----------|-------------|---------------|
| S1     | VLAN 1    | 10.10.10.2  | 255.255.255.0 |
| PC1    | NIC       | 10.10.10.10 | 255.255.255.0 |

---

## Partie 1 : Sécuriser les mots de passe

### a. Connexion initiale via Telnet

Depuis l'invite de commandes de **PC1**, on se connecte au switch S1 via Telnet.  
Telnet est utilisé ici uniquement parce que SSH n'est pas encore configuré.

```
telnet 10.10.10.2
```

- Mot de passe User EXEC : `cisco`
- Mot de passe Privileged EXEC : `cisco` (après `enable`)

### b. Sauvegarder la configuration

Avant de faire des modifications, on sauvegarde la config actuelle.  
Ainsi, si une erreur survient, il suffit de redémarrer S1 pour revenir à cet état.

```
S1> enable
S1# copy running-config startup-config
```

### c. Vérifier que les mots de passe sont en clair

On affiche la configuration pour constater que les mots de passe sont lisibles en clair — ce qui est un problème de sécurité.

```
S1# show running-config
```

On voit par exemple : `password cisco` — le mot de passe est visible par n'importe qui ayant accès au fichier.

### d. Activer le chiffrement des mots de passe

La commande `service password-encryption` chiffre tous les mots de passe en clair présents dans la configuration (type 7).

```
S1# conf t
S1(config)# service password-encryption
```

### e. Vérifier le chiffrement

On relance `show running-config` pour confirmer que les mots de passe apparaissent maintenant sous forme chiffrée (ex: `7 08026F6028`).

```
S1(config)# end
S1# show running-config
```

## Partie 2 : Chiffrer les communications (configurer SSH)

### Étape 1 – Définir le nom de domaine et générer les clés RSA

#### a. Configurer le nom de domaine

SSH a besoin d'un nom de domaine pour générer les clés cryptographiques. Sans cette étape, la commande `crypto key generate rsa` ne fonctionnera pas.

```
S1# conf t
S1(config)# ip domain-name netacad.pka
```

#### b. Générer les clés RSA

On génère une paire de clés RSA qui servira à chiffrer les communications SSH.  
Une longueur de **1024 bits** est demandée dans cet exercice (en production, 2048 bits est recommandé).

```
S1(config)# crypto key generate rsa
```

Quand le système demande la taille de la clé :

```
How many bits in the modulus [512]: 1024
```

---

### Étape 2 – Créer un utilisateur SSH et configurer les lignes VTY

#### a. Créer un utilisateur local

On crée un compte local `administrator` avec le mot de passe `cisco`.  
Le mot-clé `secret` stocke le mot de passe en MD5 (plus sécurisé que `password`).

```
S1(config)# username administrator secret cisco
```

#### b. Configurer les lignes VTY pour SSH uniquement

Les lignes VTY (Virtual Terminal Lines) gèrent les connexions à distance (Telnet, SSH).  
On les reconfigure pour :
- utiliser la base de données locale pour l'authentification (`login local`)
- n'accepter que les connexions SSH (`transport input ssh`)
- supprimer l'ancien mot de passe de ligne (`no password`)

```
S1(config)# line vty 0 15
S1(config-line)# login local
S1(config-line)# transport input ssh
S1(config-line)# no password
S1(config-line)# exit
```

---

## Partie 3 : Vérifier l'implémentation SSH

### a. Tenter une connexion Telnet (doit échouer)

On quitte la session actuelle et on essaie de se reconnecter en Telnet.  
La connexion doit être **refusée**, ce qui confirme que Telnet est bien désactivé.

```
S1(config)# end
S1# exit

(depuis PC1)
telnet 10.10.10.2
→ Connection refused  ✅
```

### b. Se connecter en SSH

On utilise maintenant SSH pour se connecter.  
L'option `-l` (lettre L minuscule, pas le chiffre 1) permet de spécifier le nom d'utilisateur.

```
ssh -l administrator 10.10.10.2
```

- Mot de passe : `cisco`

> 💡 Si tu tapes `ssh` sans paramètre et appuies sur Entrée, Packet Tracer affiche l'aide avec la syntaxe complète.

### c. Sauvegarder la configuration finale

Une fois connecté en SSH, on passe en mode privilégié et on sauvegarde la configuration.

```
S1> enable
S1# copy running-config startup-config
```

---

## Résumé des commandes clés

| Commande | Rôle |
|----------|------|
| `service password-encryption` | Chiffre les mots de passe en clair dans la config |
| `ip domain-name <nom>` | Définit le nom de domaine (requis pour RSA) |
| `crypto key generate rsa` | Génère les clés RSA pour SSH |
| `username <user> secret <mdp>` | Crée un utilisateur local sécurisé |
| `login local` | Utilise la base locale pour l'authentification VTY |
| `transport input ssh` | Autorise uniquement SSH sur les lignes VTY |
| `no password` | Supprime le mot de passe de ligne VTY |
| `ssh -l <user> <ip>` | Se connecte en SSH depuis un terminal |

---

> ⚠️ **En cas de problème** : Si l'accès à S1 est impossible, éteignez et rallumez S1 dans Packet Tracer pour revenir à la configuration sauvegardée, puis recommencez depuis la Partie 1.

## Script complet (toutes les commandes dans l'ordre)
 
```
! ─── Depuis PC1 ───────────────────────────────────────────
telnet 10.10.10.2
 
    ! ─── User EXEC (mot de passe : cisco) ─────────────────
    enable
 
        ! ─── Privileged EXEC (mot de passe : cisco) ───────
        copy running-config startup-config
        show running-config
 
        configure terminal
 
            ! ─── Global Config ────────────────────────────
            service password-encryption
 
            ip domain-name netacad.pka
            crypto key generate rsa
                ! → taille de clé : 1024
 
            username administrator secret cisco
 
            line vty 0 15
                ! ─── VTY Config ───────────────────────────
                login local
                transport input ssh
                no password
                exit
 
            end
 
        show running-config
        exit
 
! ─── Telnet refusé à partir d'ici ─────────────────────────
telnet 10.10.10.2
    ! → Connection refused ✅
 
! ─── Connexion SSH ─────────────────────────────────────────
ssh -l administrator 10.10.10.2
 
    ! ─── User EXEC (mot de passe : cisco) ─────────────────
    enable
 
        ! ─── Privileged EXEC ───────────────────────────────
        copy running-config startup-config
```
